DOC-tor Documentation

Pretty and good docs

Introduction

Welcome to the DOC-tor documentation. This guide is a meta-documentation (documentation for the documentation tool. Pretty cool, huh?)

DOC-tor is a lightweight frontend framework designed for app documentation.

The main goal of DOC-tor is to help developers to write Html documentation for projects without the need to use a gigantic frontend framework with a steep learning curve.

Through DOC-tor, a developer can write docs that will open in mobile devices as well as astonishing resolution TVs and Monitors.

DOC-tor uses no classes or ids for styling, relying only on HTML tags.

This way, you can write your docs without thinking too much about styling, cutting off development time.

DOC-tor aims at developers who want to write docs but have tight deadlines.

Installation

You can use DOC-tor by placing this line of code into your doc.html file (you can name it whatever you want, it doesn't matter). The code: <link rel="stylesheet" href="http://ulissescaon.com/style.css">

That's it. You're good to go!

Requirements

DOC-tor uses Sass for everything. If you want to modify DOC-tor, you may install Sass. But remember, if you just want to use it, there is no need for installation.

Who is DOC-tor for?

Lazy coders who feel the need to write pretty docs without devoting their lives to do that.

Sample Code

As DOC-tor uses no classes or ids and rely exclusively on html5 tags, here's a simple emmet oneliner:

!>(nav>ol>li*5)+(main>(header>h1{APP name}+h2{Description})+(article>p{content})+(footer>button))

This single line of code creates a sidebar with 5 ordered list-item elements, a main section with header and the article element, holding the content. The footer holds, by default, buttons.

One tab key away from heaven!

If you don't use Emmet, you can use this structure:

  • body
    1. nav
    2. header
      1. h1
      2. h2
    3. main
      1. article
        1. p
        2. code
      2. footer
        1. button

Each word represents a tag. You put your app name and description inside a header tag, your ordered or unordered list inside a nav tag and everything from the docs text inside the article tag.

The footer is holding buttons, but may hold your info!

If you use ids on your headers, you can link your docs and create an easy index, like this very page.
